## 1. Ключевая концепция

**Эволюция: Code → Low-code → No-code → AI-generated services.**

Вся услуга — один декларативный, версионируемый JSON-документ (**Service Definition**). Единый рантайм-движок рендерит любую услугу. Есть три способа создать Definition:

1. **AI-генератор**: администратор вставляет текст программы господдержки (или загружает документ) → LLM возвращает готовый Service Definition → администратор дорабатывает в конструкторе → публикует.
2. **Визуальный конструктор**: drag-and-drop сборка шагов/полей/правил.
3. **Импорт JSON** (для экспертов и миграций).

Масштабирование на 70+ услуг = 70 JSON-документов, не 70 разработок. Новая дочерняя организация = новая запись `organizations`. Новая интеграция = новый коннектор к mock-ЕИШ.

## 3. Модель данных

### 3.1. Таблицы (PostgreSQL)

- `organizations` — дочерние организации (Даму, БРК, КАФ, АКК, KazakhExport, Отбасы, КЖК, QIC + Холдинг). Поля: id, name, short_name, logo, color, site_url.
- `users` — id, iin_bin, role (`entrepreneur` | `admin` | `author`), profile JSONB. Авторизация: mock eGov IDP (см. §8).
- `service_definitions` — **ядро системы**: id, org_id, slug, status (`draft`/`published`/`archived`), version (int), definition (JSONB), created_by, published_at. Версионирование: каждая публикация — новая строка version+1; заявки ссылаются на конкретную версию.
- `applications` — заявки: id, number (`EPPB-2026-000001`), user_id, service_id + service_version, status, data JSONB (ответы), stage_index, timeline JSONB (история событий), created/updated.
- `documents` — файлы заявок: id, application_id, field_key, filename, mime, size, storage_path, signed_by_ecp (bool), signature_meta JSONB.
- `dictionaries` — справочники: code, name, items JSONB (КАТО-регионы, ОКЭД-отрасли, виды скота, типы вагонов, банки-партнеры и т.п.).
- `notifications` — id, user_id, application_id, type, title, body, read_at.
- `analytics_materials` — каталог модуля 6.7: id, org_id, type (`dashboard`/`report`/`financial`/`research`), title, description, source, period, url, embed_allowed (bool).
- `map_projects` — id, org_id, name, region_code, locality, lat, lng, industry, amount, period_start/end, status, description. **Все сид-данные помечены `is_demo=true` и в UI подписаны «демонстрационные данные»** — не выдаем сгенерированное за реальное.
- `knowledge_items` — модуль 6.9: id, category (`guide`/`template`/`checklist`/`calculator`/`review`), title, description, content/url.
- `audit_log` — кто/что/когда в админке (аргумент про администрируемость + доверие).

### 3.2. Service Definition (DSL) — сердце проекта

Pydantic-схема + JSON Schema в `docs/service-definition.md`. Структура:

```jsonc
{
  "meta": {
    "title": "Приобретение вагонов в лизинг — I этап",
    "org": "kdb-leasing",
    "category": "leasing",
    "audience": ["ЮЛ", "ИП"],
    "summary_plain": "Простое описание своими словами (генерирует AI)",
    "conditions": [ { "label": "Сумма", "value": "до ... ₸" } ],
    "documents_checklist": ["Справка о гос. регистрации", "..."],
    "result": "Решение о финансировании / переход ко II этапу",
    "sla_days": 15
  },
  "stages": [                     // ← многоэтапность услуги (этап = отдельная сессия подачи)
    {
      "key": "initial",
      "title": "Первичная заявка",
      "steps": [                  // ← шаги внутри этапа (страницы мастера)
        {
          "key": "applicant",
          "title": "Заявитель",
          "fields": [
            {
              "key": "bin", "type": "bin",
              "label": "БИН", "required": true,
              "prefill": "integration:gbd_ul.bin",     // предзаполнение из mock-ГБД ЮЛ
              "validate": ["bin_checksum"],
              "hint": "Подтянем данные компании автоматически"
            },
            {
              "key": "applicant_type", "type": "select",
              "dictionary": "applicant_types"
            }
          ]
        }
      ]
    },
    { "key": "extended", "title": "Расширенные данные и документы", "steps": [ "..." ] }
  ],
  "rules": [                      // видимость / обязательность / переходы
    { "if": {"field": "applicant_type", "eq": "ip"},
      "then": {"hide": ["corporate_block"], "require": ["ip_certificate"]} }
  ],
  "computed": [                   // автоматические расчеты (критерий 9.2 — 3 балла)
    { "key": "monthly_payment",
      "formula": "annuity(amount, rate, term_months)",
      "display": "Предварительный платеж: {value} ₸/мес" }
  ],
  "statuses": {                   // статусная модель, отображаемая в ЛК
    "flow": ["submitted", "registered", "in_review_bpm", "docs_requested",
             "approved", "rejected"],
    "labels_plain": { "in_review_bpm": "Заявку рассматривает Даму" }
  },
  "integrations": ["egov_idp", "gbd_ul", "ecp_sign", "bpm_submit", "bpm_status"]
}
```

Поддерживаемые типы полей (минимум): `text`, `textarea`, `number`, `money`, `select`, `multiselect`, `radio`, `checkbox`, `date`, `iin`, `bin`, `phone`, `email`, `file`, `repeater` (повторяемые группы — напр., список вагонов / поголовье), `info` (подсказка-блок), `computed` (read-only результат формулы).

**Движок формул:** безопасный evaluator (whitelist функций: арифметика, `annuity`, `min/max/round`, `if`). Никакого `eval`.

### 3.3. Движок (`backend/app/engine/`)

- `renderer_contract.py` — выдает фронту нормализованное описание текущего шага (с учетом rules и уже введенных данных): какие поля видимы, что обязательно, вычисленные значения.
- `validator.py` — серверная валидация ответов по Definition (двойная: клиент + сервер).
- `rules.py` — интерпретатор условий (eq/ne/gt/in/contains, and/or).
- `formulas.py` — вычисления.
- `lifecycle.py` — статусная машина заявки, генерация номера, таймлайн, уведомления.
- Покрытие тестами движка — приоритет №1 (это и есть 10 баллов за код).

## 4. Публичный контур (frontend, критерий 9.9 — 20 баллов)

Основной публичный вход клиентского контура: `https://bizdnai.com/baiterek/take`.

### 4.1. Главная страница
Антитеза baiterek.gov.kz. Один экран отвечает на 4 вопроса: что здесь есть / с чего начать / как подать / где мои заявки.

- Hero: строка **умного подбора** — «Опишите свой бизнес или задачу своими словами» (AI-подбор меры, §7.1) + примеры запросов чипсами («ферма 200 голов в СКО», «купить вагоны в лизинг», «нужна гарантия по кредиту»).
- Навигация по направлениям (6–8 плиток): Кредитование · Лизинг · Гарантии · Субсидии · Экспорт · Жилье · АПК · Инвестиции.
- Популярные услуги (карточки с плашкой организации).
- Блок «Как это работает» — 4 шага: найдите меру → подайте онлайн → отслеживайте статус → получите результат.
- Быстрые входы: Личный кабинет · Карта проектов · Аналитика · Инструменты для бизнеса.
- Без оргструктуры Холдинга на первом экране. Дочерние организации — как бренды-поставщики услуг внутри карточек, не как навигация.

### 4.2. Каталог и карточка услуги
- Каталог: поиск, фильтры (направление, организация, аудитория ЮЛ/ИП, сумма), сортировка.
- Карточка услуги (из `meta` Definition): человеческое описание, условия таблицей, чек-лист документов, срок, результат, кнопка «Подать заявку», блок «Объяснить простыми словами» (AI, §7.1).

### 4.3. Рантайм подачи заявки (form-engine)
- Пошаговый мастер: прогресс-бар этапов и шагов, автосохранение черновика на каждом шаге, возврат назад без потери данных.
- Предзаполнение из mock-ГБД ЮЛ по БИН (кнопка «Заполнить автоматически»).
- Инлайн-валидация, человеческие тексты ошибок, подсказки у полей.
- Экран проверки перед отправкой + AI-проверка полноты (§7.1).
- Подписание mock-ЭЦП → отправка → экран успеха: номер заявки, что дальше, кнопка в ЛК.
- Многоэтапность: после «одобрения I этапа» (переключается админом/BPM-mock) пользователь получает уведомление и продолжает этап II в той же заявке.

### 4.4. Личный кабинет (6.10)
Список заявок (номер, услуга, организация, статус-бейдж, дата) → детальная: таймлайн статусов, документы (мои + от организации), уведомления, запросы на дозагрузку документов, кнопка «Продолжить этап».

### 4.5. Модуль аналитической отчетности (6.7 — 3 балла)
Каталог материалов с фильтром по организации и типу; карточка: описание, источник, период актуальности; открытие по ссылке или embedding (iframe для разрешенных). Сид: годовые отчеты Холдинга, dashboards дочек (ссылки на реальные публичные ресурсы + 1–2 embedded примера).

### 4.6. Интерактивная карта проектов (6.8 — 3 балла)
- Leaflet, GeoJSON областей РК, choropleth-подсветка по количеству проектов + кластеры маркеров.
- Фильтры: организация, регион, отрасль, статус, период финансирования.
- Клик по проекту → карточка со всеми полями из ТЗ (наименование, организация, регион, отрасль, сумма, период, статус, описание) + ссылка «Подробнее».
- Панель-сводка: всего проектов / сумма / распределение по регионам (мини-бар-чарт).
- Данные: сид ~120–150 демо-проектов, распределенных правдоподобно по регионам и дочкам, с пометкой «демонстрационные данные»; в ARCHITECTURE.md — описание коннектора к ИС Аналитического центра как целевой интеграции.

### 4.7. Инструменты и материалы (6.9 — 3 балла)
Раздел с категориями: база знаний · шаблоны документов · чек-листы · калькуляторы · обзоры. Минимум два **работающих** калькулятора (аннуитетный платеж по лизингу; субсидируемая ставка) — переиспользуем движок формул из конструктора, что подчеркиваем на питче.

## 5. Административный контур (6.3, 6.11; критерии 9.2 + 9.4)

Основной публичный вход конструктора услуг: `https://bizdnai.com/baiterek/create`.

### 5.1. Реестр услуг
Таблица Definitions: статус, версия, организация, автор, дата; действия: редактировать, дублировать, опубликовать/снять, история версий, экспорт/импорт JSON.

### 5.2. Визуальный конструктор
Три панели: структура (этапы→шаги→поля, drag-and-drop) · холст-превью (живой рендер того же form-engine, что видит клиент) · свойства выбранного элемента.

- Редактор поля: тип, label, hint, required, справочник, prefill-источник, валидации.
- Редактор правил: конструктор условий «Если [поле] [оператор] [значение] → скрыть/показать/сделать обязательным/перейти» — без кода, выпадающими списками.
- Редактор расчетов: формула + предпросмотр результата на тестовых значениях.
- Кнопка «Посмотреть глазами клиента» — мгновенный превью-режим.
- Публикация: валидация целостности Definition → version+1 → услуга сразу в каталоге.

### 5.3. AI-генератор услуги (★ главный дифференциатор, §7.2)
Отдельный экран «Создать услугу из документа»: textarea/загрузка описания программы → прогресс генерации → результат открывается **в конструкторе** как draft для доработки. Диф между «сырой AI-версией» и финальной ручной правкой — показать на демо.

### 5.4. Заявки и справочники
- Реестр заявок: фильтры, смена статуса (имитация решения BPM), запрос дополнительных документов, комментарий заявителю → триггерит уведомление в ЛК.
- CRUD справочников.
- Управление контентом: аналитические материалы, проекты карты, knowledge items.
- Роли: `admin` (всё), `author` (конструктор своей организации). Audit log на все изменения.

## 6. Демонстрация конструктора (страховка от дисквалификации)

1. Обе контрольные услуги собраны в конструкторе. Услуга «Агробизнес: животноводство» дополнительно **пересобирается вживую AI-генератором на питче** (или screencast 60–90 сек в презентации).
2. В README — раздел «Как убедиться, что услуги не захардкожены»: инструкция экспертам изменить поле в конструкторе и увидеть изменение на клиенте без деплоя.
3. `make seed` создает Definitions через тот же публичный API конструктора, а не прямыми INSERT.

## 7. AI-компонент (6.6, критерий 9.4 — 15 баллов)

Не чат-бот (у Холдинга он есть — не дублируем, это оговорено в ТЗ). AI = **встроенный копайлот в конкретных точках пути**. Все AI-ответы помечены бейджем «AI» с дисклеймером; соответствие Закону РК №230-VIII (прозрачность, объяснимость, человеческий контроль — финальные решения только за администратором).

### 7.1. Для предпринимателя
- **Подбор меры** (главная): свободный текст → LLM с каталогом услуг в контексте → топ-3 меры с объяснением «почему подходит» + уточняющий вопрос при неоднозначности.
- **«Объяснить простыми словами»** (карточка услуги): пересказ условий без канцелярита + «что мне понадобится».
- **Проверка полноты заявки** (перед отправкой): анализ ответов → список «Рекомендуем проверить: …» (не блокирует отправку).
- Подсказки при заполнении сложных полей (по hint + контексту услуги).

### 7.2. Для администратора/автора
- **Генератор услуги из документа** (§5.3): промпт со строгой JSON-схемой Definition, серверная валидация результата Pydantic-схемой; при ошибке — авторепромпт (до 2 ретраев), затем честная ошибка.
- **Предложение ветвлений**: «Судя по полю "Тип заявителя", вероятно нужны разные документы для ИП и ТОО — добавить правило?»
- Автогенерация подсказок и plain-language описаний к полям.

### 7.3. Техконтур
`LLMProvider` интерфейс → `AnthropicProvider` / `MockLLMProvider` (ENV `LLM_PROVIDER`). Все промпты в `backend/app/ai/prompts/` как версионируемые файлы. Контрактные тесты: mock-провайдер возвращает эталонные ответы, тесты проверяют пайплайн генерации end-to-end без сети. Логирование AI-вызовов в audit_log (без персональных данных в логах — Закон №326-VIII).

**Бюджет LLM на публичном стенде.** Стенд открыт весь конкурс, поэтому кроме rate limit действует суточный потолок расходов (ENV `LLM_DAILY_BUDGET`, счётчик вызовов/токенов в БД). При исчерпании — graceful-фолбэк на `MockLLMProvider` с честной пометкой в UI («AI временно в демо-режиме»); функциональность портала не деградирует.

# Обязательное расширение — Плавный сбор анкетных данных и checkpoint-модель

> Дополняет SPEC.md (§3.3 движок, §4.3 рантайм, §7.1 AI). Правила ниже — обязательные
> инварианты UX, проверяются перед демо по чек-листу в конце файла.

## 1. Принцип: анкета — это диалог, а не стена полей

**Правило 3–6.** На одном экране — от 3 до 6 элементов ручного ввода, объединённых одной темой. Если после применения правил видимости на шаге остаётся больше — шаг автоматически дробится рендерером (Definition может задавать `max_fields_per_screen`, по умолчанию 6). Каждая новая страница гарантированно сохраняет все поля, заполненные ранее. Если в смысловом блоке объективно меньше трёх полей, его не дополняют искусственными вопросами.

**Порядок вопросов — от отсекающих к детальным.** Первыми идут поля-ветвители
(тип заявителя, направление, цель): каждый ответ сокращает оставшуюся анкету.
Пользователь никогда не видит вопрос, который его не касается.

**Один смысловой блок = один экран.** Заголовок экрана отвечает на вопрос «о чём
сейчас спрашиваем и зачем» («Расскажите о вашем хозяйстве — это определит доступную
сумму»). Никаких экранов «Прочие сведения».

**Сначала лёгкое, тяжёлое — в конце.** Выбор из списков и короткие ответы — в начале
пути; загрузка документов — последним блоком перед проверкой. Пользователь успевает
«вложиться» в заявку до самого трудного шага (меньше бросаний).

**Ничего не спрашиваем, если можем узнать сами.** Иерархия источников значения поля:
1) профиль/ГБД ЮЛ по ИИН/БИН (детерминированные коннекторы),
2) ответы AI-диалога подбора (см. §3),
3) вычисление формулой (computed),
4) и только потом — ручной ввод.
Предзаполненные поля показываются свернуто строкой «Данные компании: ТОО «Демо»,
БИН …, Алматы — Изменить», а не как поля для ввода.

## 2. Checkpoint-модель: «сохранить уровень и продолжить»

**Автосохранение непрерывное.** Каждое изменение поля сохраняется в черновик
(debounce 800 мс, PATCH), плюс гарантированное сохранение при переходе между
экранами и при потере фокуса вкладки. Кнопки «Сохранить» нет — сохранение не
действие пользователя, а свойство системы. Индикатор: «Черновик сохранён · 14:32». Каждый PATCH передаёт номер ревизии черновика. Сервер не позволяет запоздавшему старому запросу затереть более новое значение и возвращает актуальную ревизию.

**Модель данных.** `applications.status = 'draft'` до отправки; позиция хранится как
`checkpoint: {stage_key, step_key, screen_key}`, где `screen_key` — ключ первого поля
экрана в детерминированном разбиении (НЕ порядковый индекс: динамическое дробление
3–6 и правила видимости меняют число экранов, индекс «уезжает», ключ поля — нет;
если поле стало невидимым, рендерер позиционирует на ближайший существующий экран);
ответы — в `data` JSONB инкрементально.
Черновик один на пользователя на услугу (повторный вход в услугу = продолжение, не новая
заявка; «начать заново» — явное действие с подтверждением). Уникальный ключ активного черновика: пользователь + услуга + версия Service Definition.

**Возврат с любого устройства.** В личном кабинете черновик — первым блоком:
«Заявка "Агробизнес: животноводство" заполнена на 60% — Продолжить». Кнопка ведёт
ровно на тот экран, где пользователь остановился. Начал на телефоне — продолжил
с компьютера: состояние серверное, не localStorage.

**Прогресс как уровни.** Индикатор из названий смысловых блоков (Заявитель →
Хозяйство → Цель → Расчёт → Документы), пройденные — с галочками, по ним можно
вернуться назад без потери введённого. Проценты — вторичны, названия уровней — главное:
пользователь всегда понимает, где он, что уже сделано и сколько осталось (требование 6.2 ТЗ).

**Напоминание о черновике.** Если черновик не трогали 48 часов — уведомление в ЛК
(«Ваша заявка почти готова — осталось 2 шага»). TTL черновика — 30 дней, за 5 дней
до удаления — предупреждение. (В MVP — внутренние уведомления; email/SMS — контракт-заглушка.)

**Пересечение с версионированием.** Если услуга обновилась, пока черновик спал:
черновик продолжает жить на своей версии Definition (заявки привязаны к версии — SPEC §3.1).
Миграция черновика на новую версию — не делаем в MVP, фиксируем как известное решение.

## 3. AI-диалог как источник анкеты (связка с подбором меры)

Диалог подбора («хочу выращивать коров») — не отдельный чат, а **начало заполнения**:
- уточняющие вопросы AI (вид скота, поголовье, регион, пастбища) мапятся на ключи полей
  Definition через `meta.intake_mapping`;
- при переходе «Подать заявку» создаётся черновик с уже заполненными этими полями;
- первый экран мастера показывает сводку: «Из нашего разговора я уже заполнил: вид —
  КРС, поголовье — 200, регион — СКО. Проверьте и продолжим» — эффект «заявка наполовину
  готова ещё до её начала»;
- каждое AI-заполненное значение помечено и редактируемо; в data хранится источник
  значения (`source: ai_intake | profile | computed | manual`) — это же пригодится
  для журнала и объяснимости (№230-VIII).
- Структурированный результат AI обязательно валидируется по JSON Schema Service Definition; невалидный ответ не записывается в заявку и отправляется на повторную генерацию либо ручную проверку.

## 4. Микро-правила интерфейса шага

- Валидация — инлайн по мере ввода, человеческим языком («БИН должен содержать
  12 цифр — сейчас 11»), без красных простыней после нажатия «Далее».
- Ошибки не блокируют сохранение черновика — блокируют только переход вперёд.
- Подсказка поля — рядом, раскрывается по «?», не отдельной страницей инструкции.
- Repeater-группы (вагоны, поголовье) — добавление строк по одной, с копированием
  предыдущей строки как шаблона.
- Каждый computed-результат объясняет себя одной строкой («Платёж рассчитан по ставке
  6% на 60 месяцев»).
- Кнопка «Назад» никогда не теряет данные. Закрытие вкладки никогда не теряет данные.

## 5. API (для Claude Code)

- `POST /api/v1/applications` `{service_id}` → создаёт draft на текущей published-версии
  (или возвращает существующий активный draft **любой** версии — если услуга успела
  обновиться до v2, живой draft на v1 возвращается, второй draft на v2 не создаётся)
- `PATCH /api/v1/applications/{id}/draft` `{data_delta, checkpoint}` → инкрементальное
  сохранение; ответ — актуальные computed-значения и видимость полей следующего экрана
- `GET /api/v1/applications/{id}/resume` → Definition-версия + data + checkpoint +
  нормализованный текущий экран (то, что рендерит form-engine)
- `POST /api/v1/applications/{id}/submit` → серверная валидация всей анкеты → статусная машина
- `POST /api/v1/intake/{service_id}/handoff` `{dialog_answers}` → создание черновика из AI-диалога

## 6. Чек-лист приёмки перед демо

- [ ] Ни на одном экране обеих контрольных услуг нет больше 6 полей ручного ввода
- [ ] Закрыл вкладку на шаге 3 → вошёл с другого браузера → продолжил с шага 3
- [ ] Данные по БИН подтянулись и показаны сводкой, а не пустыми полями
- [ ] Ответы AI-подбора попали в черновик и помечены как предзаполненные
- [ ] Прогресс-бар показывает названия блоков, возврат назад не теряет данные
- [ ] В ЛК черновик виден с процентом заполнения и кнопкой «Продолжить»
- [ ] «Начать заново» запрашивает подтверждение

## 7. Контроль подтверждений, файлов и требований

Все пометки `[VERIFY]` ведутся в едином реестре: требование, проверяемое значение, официальный источник, ответственный, статус и дата проверки. Неподтверждённая цифра не публикуется как факт.

Для пользовательских загрузок проверяются разрешённое расширение, реальный MIME-тип и максимальный размер. Файл получает безопасное серверное имя и хранится вне публичного web-каталога. Полноценный антивирусный сервис не является обязательным для MVP и может быть подключён позднее.

Внутри проекта ведётся матрица трассировки `требование → тест → публичный сценарий`. Она нужна команде для контроля полноты реализации и не выводится пользователю.
