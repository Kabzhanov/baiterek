# Контрольные услуги (SPEC.md §9, docs/IMPLEMENTATION_PLAN.md §10 «Этап 5»)

Обе контрольные услуги существуют **только** как записи `service_definitions`,
созданные через публичный import/publish API (см. `backend/app/seed/control_cases.py`),
— ни в одном файле `backend/app/api`, `backend/app/engine`, `backend/app/schemas` или
`frontend` нет отраслевых наименований этих двух услуг (SPEC.md §0, проверяется
`make lint-hardcode`). Сами Definition-JSON — данные, а не код —
лежат в `backend/app/seed/fixtures/control_case_a.json` и `control_case_b.json`
(эта директория явно исключена из `lint-hardcode`-грепа).

Ниже — «Услуга A» и «Услуга B» без сокращения имени фикстур/файлов, чтобы документ
оставался читаемым; сами отраслевые термины уместны здесь по правилу задания (docs
исключён из грепа).

## Источники

- Услуга A (сложная, `control_case_a.json`, слаг `wagons-leasing`): `bgov.kz/ru/services/wagons_ind`.
- Услуга B (популярная, `control_case_b.json`, слаг `agro-livestock`): `bgov.kz/ru/services/agroanimal2`.

На момент разработки прямой сетевой доступ к обеим страницам bgov.kz из окружения
агента недоступен (см. `docs/verify-register.md`) — анкеты воссозданы по публично
известной структуре программ финансового лизинга подвижного состава и программ
кредитования животноводства через институты развития Холдинга (SPEC.md §9: «если
точные условия недоступны — поля восстанавливаем по публичным правилам программ с
пометкой в документации»). Все числовые условия (ставка, аванс, предельная сумма,
сроки, SLA) либо не задаются вовсе, либо запрашиваются у самого заявителя как ввод
формы, либо помечены строкой «по условиям программы» — см. реестр `docs/verify-register.md`
(`V-A-*`, `V-B-*`). Ни одно из этих значений не используется как блокирующая
валидация (`minimum`/`maximum` в схеме — только санитарные, не бизнес-лимиты).

## Что восстановлено по публичным правилам (а не по первичному источнику)

- **Держатели программ.** БРК — для лизинга подвижного состава (V-A-04), КАФ — для
  животноводства (V-B-06). Оба назначения — инфраструктурный выбор для демо
  (нужен реальный `org_id`), не факт, вычитанный со страницы услуги.
- **Состав документов** по типу заявителя (ИП/ТОО, КХ/ИП/ТОО/СПК) — стандартный для
  форм господдержки МСБ/АПК набор (регистрационный документ + устав для юрлиц),
  без утверждения о его полноте (V-A-03, V-B-04).
- **Регион (КАТО).** Схема DSL v1 (`backend/app/schemas/definition.py`) не имеет
  атрибута «источник — справочник» у `SelectField` (только инлайновый `options:
  list[str]`), поэтому список регионов в `control_case_b.json` — те же 20 подписей,
  что в `backend/app/seed/data.py`'s `kato_regions` (единый источник истины по
  тексту, не по ссылке в рантайме). Отмечено как техническое ограничение DSL v1,
  не как содержательный [VERIFY].
- **Вид скота.** Аналогично — инлайновый список вместо отдельного справочника (в
  фикстуре, не в общих `DICTIONARIES`), поскольку схема не поддерживает
  межфикстурные ссылки на кастомные справочники в MVP.

## Услуга A — структура (`wagons-leasing`)

Этап I (`stage-1`, 5 шагов) → индикативное одобрение → Этап II (`stage-2`, 2 шага,
все поля необязательны — см. «Ограничение движка» ниже):

1. `applicant` — тип (ИП/ТОО), наименование, БИН/ИИН, телефон.
2. `deal_scope` — состояние (новые/б.у.), тип, количество, цена за единицу, аванс %,
   срок лизинга. 6 полей — гейтинг-поля идут первыми.
3. `wagon_details` — repeater позиций + условные поля: гарантия (только «новые»),
   год выпуска и номер акта осмотра (только «б.у.») — ветвление «новые/б.у. → доп.
   поля» из SPEC §9.
4. `financing` — индикативная ставка (ввод заявителя, не хардкод), участие в
   субсидировании (условно раскрывает номер сертификата).
5. `documents` (последний шаг) — документ ИП или устав ТОО (взаимоисключающее
   ветвление «ИП/ТОО → разный состав документов»), финотчётность, доверенность.

Computed (`total_cost → advance_amount → financing_amount → monthly_payment`,
топологический порядок вычисляется движком): сумма сделки, аванс, сумма
финансирования и аннуитетный платёж с трассировкой (`explanations.computed` в ответе
рендера) — ни одно значение не захардкожено, все — из введённых заявителем чисел.

Статусная модель: `draft → submitted → in_review_bpm → indicative_approved |
rejected`. `indicative_approved` — «этап II открывается по одобрению» из SPEC §9.

## Услуга B — структура (`agro-livestock`)

Этап I (`stage-1`, 7 шагов) → Этап II (`stage-2`, 1 шаг, дозагрузка документов):

1. `applicant` — тип заявителя КХ/ИП/ТОО/СПК, наименование, БИН/ИИН, телефон.
2. `farm` — регион (КАТО), район, площадь участка, право пользования землёй.
3. `livestock` — вид скота, племенное направление, repeater поголовья.
4. `purpose` — цель финансирования (приобретение/строительство/оборотка) + сумма.
5. `purpose_details` — по одному полю на каждую цель, видимо только своё
   («ветвление по цели» из SPEC §9).
6. `financing` — стоимость обеспечения (0 допустим), участие в господдержке
   (условно раскрывает номер программы).
7. `documents` (последний шаг) — документ по типу заявителя (КХ | ИП | ТОО/СПК —
   взаимоисключающие ветви), подтверждение прав на землю.

Computed: `collateral_coverage_ratio = round(collateral_value / requested_amount, 2)`
— прозрачный коэффициент из введённых заявителем чисел (не бизнес-лимит).
«Предельная сумма/ставка — по условиям программы» — сознательно **не** реализовано
как формула с выдуманной константой (см. «Ограничение движка» ниже, пункт про
`if`/boolean); вместо этого это статический `meta.conditions` (тот же контракт,
что уже использует `backend/app/seed/data.py` для демо-услуг и что публично
показывает `GET /services`), поэтому текст никогда не участвует в блокирующей
валидации.

Статусная модель: `draft → submitted → in_review_bpm → document_request | approved |
rejected` — «этап II — дозагрузка документов по запросу».

## Ограничение движка, важное для дизайна обеих фикстур

`backend/app/engine/formulas.py`'s `evaluate_formula()` конвертирует любое
`$`-значение через `Decimal(str(data[key])) if isinstance(data[key], (int, float,
Decimal))`; `bool` — подкласс `int` в Python, поэтому булево поле, использованное
внутри `computed`-выражения, ловит `Decimal(str(True))` → `decimal.InvalidOperation`
на каждой попытке рассчитать (submit возвращает 422 `computation_failed` вместо
результата). В `backend/app/engine/rules.py`'s `evaluate_condition()` такого бага
нет (значение передаётся как есть). Обе фикстуры поэтому **не** ссылаются на булевы
поля внутри `computed.expression` — только внутри `rules[].when`. Это ограничение
существующего движка (файл вне зоны этой задачи), не архитектурное решение фикстур;
задокументировано здесь, чтобы будущая правка `evaluate_formula()` не была
неожиданностью при следующей ревизии контрольных кейсов.
