# Архитектура ЕППБ MVP

## 1. Цель

ЕППБ объединяет услуги дочерних организаций Холдинга «Байтерек» в одном
клиентском интерфейсе и предоставляет администраторам универсальный конструктор.
Архитектурный принцип: новая услуга добавляется данными, а не отдельной разработкой.

## 2. Главный архитектурный контракт

`Service Definition` — декларативный, версионируемый JSON-документ. Он содержит:

- метаданные, условия, документы и результат услуги;
- этапы, шаги и поля;
- правила видимости, обязательности и переходов;
- безопасные вычисляемые формулы;
- статусную модель;
- ссылки на интеграционные коннекторы;
- mapping между AI-intake и полями анкеты.

Backend валидирует Definition по Pydantic/JSON Schema. Frontend не реализует
конкретные услуги, а рендерит нормализованный контракт текущего экрана.

## 3. Контуры

```text
Пользователь
    │
    ▼
/baiterek/take ──► AI-подбор ──► Form Engine ──► Заявка/черновик
                                              │
                                              ▼
                                      Lifecycle + mock ЕИШ

Администратор
    │
    ▼
/baiterek/create ─► AI Generator / Visual Constructor / JSON Import
                                              │
                                              ▼
                                  Draft Definition ─► Publish version
```

## 4. Компоненты

### Frontend

Next.js 14, TypeScript, Tailwind CSS, shadcn/ui, Zustand и react-hook-form.

- `form-engine` — тематические экраны по 3–6 ручных полей;
- `constructor` — структура, live preview и свойства элементов;
- клиентский кабинет — черновики, заявки, timeline и документы;
- публичные модули — каталог, аналитика, карта и материалы.

### Backend

Python 3.12, FastAPI, SQLAlchemy 2 async, Pydantic v2, Alembic.

- `renderer_contract` — вычисляет текущий экран с учётом rules;
- `validator` — серверная валидация Definition и ответов;
- `rules` — интерпретатор условий без исполнения произвольного кода;
- `formulas` — whitelist-функции, без `eval`;
- `lifecycle` — статусы, этапы, номера заявок, timeline и уведомления;
- `ai` — абстракция LLM, генератор Definition и copilot;
- `integrations` — единые контракты внешних систем и mock-реализации.

### Данные

PostgreSQL 16 и JSONB. Основные сущности:

- `organizations`;
- `service_definitions` с неизменяемыми опубликованными версиями;
- `applications`, привязанные к версии Definition;
- `documents`, `notifications`, `dictionaries`;
- `analytics_materials`, `map_projects`, `knowledge_items`;
- `audit_log`.

## 5. Черновики и checkpoint

- На пользователя, услугу и версию Definition существует один активный черновик.
- Ответы сохраняются инкрементально с debounce 800 мс.
- Переход на новую страницу гарантированно сохраняет ранее введённые поля.
- `checkpoint = {stage_key, step_key, screen_index}` хранится на сервере.
- Каждый PATCH содержит ревизию; старый запрос не может затереть более новый.
- Ошибки блокируют переход вперёд, но не сохранение черновика.
- Черновик продолжает работать на исходной версии Definition.

## 6. AI и объяснимость

AI используется для подбора меры, генерации Definition, проверки полноты и copilot
администратора. Каждый структурированный ответ проверяется по JSON Schema.
Невалидный ответ не публикуется и не записывается в заявку.

Для значений анкеты хранится источник:

```text
ai_intake | profile | computed | manual
```

Публикация AI-сгенерированной услуги требует просмотра и подтверждения администратором.

## 7. Интеграции

Все внешние системы подключаются через стабильные интерфейсы:

- eGov IDP;
- ГБД ЮЛ;
- ЭЦП;
- BPM;
- уведомления;
- аналитический центр.

В MVP используются детерминированные mock-коннекторы с явной маркировкой в UI.
Их можно заменить реальными адаптерами без изменения Service Definition и form-engine.

## 8. Файлы и безопасность

- allowlist расширений и реальная проверка MIME-типа;
- лимит размера;
- безопасное серверное имя;
- хранение вне публичного web-каталога;
- секреты только через ENV;
- аудит административных действий;
- отсутствие произвольного `eval` для формул и правил.

Полноценное антивирусное сканирование файлов не входит в обязательный объём MVP,
но может быть подключено отдельным адаптером.

## 9. Развёртывание

Целевой стек запускается Docker Compose на VM:

```text
nginx ─► web (Next.js)
      └► api (FastAPI) ─► PostgreSQL
                         ├► LLM provider
                         └► mock/real integration adapters
```

Nginx публикует только HTTPS-маршруты `/baiterek/take` и `/baiterek/create`.
PostgreSQL и внутренние сервисы не выставляются наружу.

## 10. Проверяемость

Для каждого обязательного требования ведётся внутренняя связь:

```text
требование → автоматический тест → публичный пользовательский сценарий
```

Отдельно ведётся реестр `[VERIFY]` для условий программ: значение, официальный
источник, ответственный, статус и дата проверки. Неподтверждённые значения не
публикуются как факты.

## 11. Следующие архитектурные артефакты

- JSON Schema Service Definition;
- OpenAPI-контракты applications/intake/constructor;
- контракты mock-интеграций;
- модель угроз и права ролей;
- схема deployment и backup/restore;
- ADR по версионированию Definition и конкурентному автосохранению.
